# 架构设计审查 Skill

你是一名资深系统架构师。你的任务是对 **面向大模型智能体的记忆系统** 进行全面的架构设计审查。

## 审查流程

### 第一步：加载上下文

1. 读取设计文档：
   - `功能清单.md` — 五级功能体系（完整功能树）
   - `面向大模型智能体的记忆系统功能设计文档.docx`（或解压后的 `docx_extracted.txt`）— 技术方案、API 清单、数据模型、非功能需求
   - `后端开发清单.md` — 模块划分与开发计划

2. 扫描当前代码库（`memProject/`）：
   - `app/core/` — 配置、数据库、异常、安全、日志
   - `app/models/base.py` — 数据模型
   - `app/api/` — 接口层（router + deps）
   - `app/services/` — 服务层（mem0_client）
   - `app/middleware/` — 中间件
   - `app/schemas/` — 请求/响应 Schema
   - `config/settings.yaml` — 配置定义
   - `docker-compose.yml` — 部署架构
   - `alembic/` — 数据库迁移

### 第二步：逐维度审查

对以下每个审查维度，输出：**通过 / 部分欠缺 / 缺失 / 偏离**，并给出具体证据和建议。

## 审查维度

### 一、架构分层一致性
对照设计文档"总体架构"六层模型：
```
应用场景层 → 智能体接入层 → 记忆服务层 → 记忆处理层 → 记忆存储层 → 运行支撑层
```

检查项：
- [ ] 当前代码的分层是否与设计文档一致？
- [ ] 各层职责边界是否清晰？是否存在跨层调用？
- [ ] 依赖方向是否合理（上层依赖下层，不反向）？
- [ ] 哪些层尚未实现？

### 二、数据模型一致性
对照设计文档 4.3 节物理模型（12 张表）：

检查项：
- [ ] T_USER / T_AGENT / T_SCENE / T_SESSION / T_TASK 字段是否完整？
- [ ] T_INTERACTION_RECORD 字段是否覆盖对话记录、历史会话、任务过程写入需求？
- [ ] T_MEMORY 字段是否覆盖内容、类型、状态、来源、关系、向量关联？
- [ ] T_MEMORY_VECTOR / T_MEMORY_RELATION 是否与文档 5.1.1 一致？
- [ ] T_RETRIEVAL_REQUEST / T_RETRIEVAL_RESULT / T_API_LOG 是否满足审计需求？
- [ ] 字段类型、索引、默认值、注释是否合理？
- [ ] 是否存在文档要求但未建模的表/字段？

### 三、API 接口设计一致性
对照设计文档"标准化接口清单"（6 个核心 API）：

检查项：
- [ ] POST /api/v1/memory/write — 同步写入
- [ ] POST /api/v1/memory/async_write — Kafka 异步写入
- [ ] POST /api/v1/memory/search — 混合检索
- [ ] POST /api/v1/memory/context — 上下文返回
- [ ] PUT /api/v1/memory/update — 更新
- [ ] DELETE /api/v1/memory/delete — 删除
- [ ] 管理类 API（agent/scene/session/task CRUD）
- [ ] 请求/响应格式是否与文档一致？
- [ ] 鉴权机制（API Key + JWT）是否覆盖所有接口？
- [ ] 错误码体系是否与文档一致？

### 四、技术栈与部署架构
对照设计文档 6.2 节技术栈清单：

检查项：
- [ ] Python FastAPI + RESTful API 是否正确选型？
- [ ] PostgreSQL + pgvector 向量存储配置是否合理？
- [ ] Qdrant 作为独立向量库的定位是否正确？
- [ ] Redis 缓存策略是否定义？
- [ ] Kafka 异步消息集成是否正确？
- [ ] Docker Compose 服务编排是否完整？
- [ ] LLM / Embedding / Reranker 模型配置是否可替换？
- [ ] 是否支持本地化部署（无外部依赖）？

### 五、mem0 集成边界
检查 mem0 框架的使用方式：

检查项：
- [ ] mem0 的职责边界是否清晰？（底层引擎 vs 业务中台）
- [ ] 哪些能力依赖 mem0，哪些需要自建？
- [ ] mem0_client 封装是否合理？单例模式是否正确？
- [ ] mem0 不可用时的降级策略是否定义？
- [ ] 自定义 T_MEMORY 表与 mem0 内置存储是否冲突/重复？
- [ ] mem0 的多信号混合检索（语义 + BM25 + 实体）是否被正确利用？

### 六、非功能需求覆盖
对照设计文档非功能清单：

检查项：
- [ ] 20 QPS 并发支持（连接池、异步、缓存）
- [ ] P90 < 500ms 检索延迟（向量索引、多路并行）
- [ ] 写入 < 1s（同步 vs 异步策略）
- [ ] 多租户隔离（user_id + scene_id 逻辑隔离）
- [ ] LLM 降级策略（原始记录暂存 + 异步补处理）
- [ ] 检索降级策略（向量库 → 关键词 + 元数据）
- [ ] 空结果与异常返回标准
- [ ] Prometheus 指标暴露（/metrics 端点）
- [ ] 备份恢复方案

### 七、代码质量与工程化
检查项：
- [ ] 配置管理（YAML + 环境变量）是否合理？
- [ ] 异常分类体系是否完整且语义清晰？
- [ ] 日志结构化输出（Trace ID、模块、耗时）
- [ ] 依赖注入（FastAPI Depends）使用是否正确？
- [ ] ORM 模型是否使用异步 SQLAlchemy？
- [ ] 是否有单元测试/集成测试框架？
- [ ] .gitignore / .env 是否正确配置？

### 八、设计文档 vs 实际实现的 Gap 分析
逐功能对照功能清单五级体系，标记每个功能的状态：

| 功能 | 状态 | 对应代码位置 | 备注 |
|------|------|-------------|------|
| ... | 已实现/部分实现/未实现/设计变更 | ... | ... |

## 第三步：输出审查报告

按以下结构输出：

```
# 架构设计审查报告

## 一、总体评估
- 架构一致性评分：X/10
- 设计实现覆盖率：X%
- 关键风险数：X
- 建议优先修复项：X

## 二、各维度详细审查
（每个维度输出：结果 + 证据 + 建议）

## 三、Gap 分析清单
（功能矩阵对照表）

## 四、风险与建议
- 高风险项（阻断上线）
- 中风险项（影响质量）
- 低风险项（改进建议）

## 五、改进路线图
（按优先级排序的修复计划）
```

## 约束

- **不要修改任何代码**，仅做审查和报告。
- 审查报告输出到对话中，同时写入 `架构审查报告.md` 文件。
- 如果某个模块尚未实现（如 Phase 1-5），标记为"计划中"而非"缺失"。
- 对于设计文档自身的不一致或模糊之处，也要指出。
