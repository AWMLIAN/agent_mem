# 项目架构详解 — 面向大模型智能体的记忆系统

> 2394 行 Python 代码 / 51 个文件

---

## 一、系统定位

这不是一个简单的 CRUD 应用。它是一套记忆处理管线：

```
智能体交互数据
    ↓
接入校验 → 原始记录入库
    ↓
LLM 抽取 → 偏好/事实/任务/决策
    ↓
向量化(Embedding) → Qdrant
    ↓
去重融合 → 结构化记忆入库
    ↓
检索时：语义 + BM25 + 元数据 → 融合排序 → Prompt片段 → 返回智能体
```

核心价值：从海量对话中自动沉淀出可复用的结构化记忆，让智能体在长周期任务中不遗忘、不重复询问。

### exceptions.py — 异常类定义

12种分类异常，每种带有错误码、HTTP状态码：

```
AppException (基类: code, message, status_code, detail)
├── ValidationError      422  参数校验失败
├── AuthenticationError  401  认证失败
├── AuthorizationError   403  权限不足
├── NotFoundError        404  资源不存在
├── ConflictError        409  资源冲突
├── MemoryWriteError     500  记忆写入失败
├── MemoryRetrievalError 500  记忆检索失败
├── MemoryGenerationError 500 记忆生成失败
├── LLMServiceError      503  大模型服务异常
├── VectorStoreError     503  向量库服务异常
├── DatabaseError        500  数据库异常
└── ServiceDegradedError 503  服务降级中
```

所有业务代码只需 `raise NotFoundError("描述")`，中间件会自动转为：
```json
{"code":-1, "message":"描述", "error_code":"NOT_FOUND", "detail":{}, "trace_id":"xxx"}
```

## 七、app/middleware/ — 中间件层（3个文件）

### logging.py — 请求日志中间件

每个HTTP请求经过它时：

1. 从请求头取 `X-Trace-Id`，没有则生成 `uuid4().hex[:16]`
2. 存入 `request.state.trace_id`
3. 记录开始时间
4. 执行后续处理
5. 计算耗时（毫秒）
6. Response头加 `X-Trace-Id` 和 `X-Response-Time`
7. 写结构化日志：`trace_id, method, path, status_code, elapsed_ms`

作用：一次请求的全链路追踪，所有日志都带同一个 `trace_id`。

### auth.py — 认证中间件

- 提取 `X-API-Key` 请求头 → 存入 `request.state.api_key`
- 提取 `Authorization: Bearer xxx` → 存入 `request.state.token`
- 公开路径（`/health`、`/docs`、`/metrics`）跳过提取
- **不拦截请求**，拦截逻辑由 `deps.py` 的依赖注入负责。这样管理后台和业务接口可定义不同的鉴权规则

### error_handler.py — 异常处理中间件

三种异常拦截：

| 异常类型 | 处理方式 |
|----------|----------|
| `AppException`（业务异常） | 返回对应的HTTP状态码+JSON，warning级别日志 |
| `Exception`（未知异常） | 返回500+隐藏错误细节（debug模式才暴露），exception级别日志 |
| 422 Validation Error | 返回参数校验失败的字段详情 |

### 核心表详解：T_MEMORY

这是整个系统最核心的表。每条记忆记录：

| 字段分组 | 字段 | 说明 |
|----------|------|------|
| 归属 | user_id, agent_id, scene_id, session_id, task_id | 这条记忆属于谁、在哪个场景、哪个会话、哪个任务中产生 |

| 分类 | memory_type | preference(偏好)/fact(事实)/task(任务状态)/process(过程)/feedback(反馈)/constraint(约束)/decision(决策) |
| 生命周期 | status, version, replaced_by | 当前是否有效、版本号、被谁替代 |

| 来源 | source_type(extracted/imported/manual), source_record_ids | 这条记忆从哪些原始交互记录中抽取 |
| 向量 | vector_id | 对应Qdrant中的向量ID，用于语义检索 |

## 十一、app/services/mem0_client.py — mem0引擎封装

整个系统最关键的120行代码。它是我们和mem0框架之间的桥梁。

### 职责划分

```
我们负责：                        mem0 负责：
- 接入鉴权                       - LLM 调用 DeepSeek 抽取记忆
- 多层记忆组织(用户/会话/任务)    - Embedding 调用 硅基流动 向量化
- 记忆状态生命周期管理             - 混合检索（语义+BM25+实体匹配）
- 记忆关系建模（冲突/替代/继承）   - 去重
- 上下文装配（Prompt片段生成）     - PG history_store 读写
- 审计日志                       - Qdrant vector_store 读写
```

### 方法说明

| 方法 | 内部调用 | 被谁调用 | Phase |
|------|----------|----------|-------|
| `initialize()` | `Memory.from_config()` 初始化mem0 | lifespan启动时 | 0 |
| `add(messages, user_id)` | `mem0_client.add()` → DeepSeek抽取 → bge-m3向量化 → PG+Qdrant写入 | `/memory/write` | 2 |
| `search(query, filters)` | `mem0_client.search()` → 语义+BM25+实体三路融合检索 | `/memory/search` | 4 |
| `update(memory_id, data)` | `mem0_client.update()` | `/memory/update` | 2 |
| `delete(memory_id)` | `mem0_client.delete()` | `/memory/delete` | 2 |
| `get(memory_id)` | `mem0_client.get()` | 管理后台 | 5 |
| `get_all(user_id)` | `mem0_client.get_all()` | 管理后台 | 5 |

### 关键设计决策

| 问题 | 决策 | 原因 |
|------|------|------|
| Qdrant 用 HTTP 还是 gRPC？ | **gRPC(6334端口)** | Qdrant v1.18 HTTP API 与 qdrant-client v1.18.0 不兼容，返回502 |
| SiliconFlow 传不传 dimensions？ | **不传** | SiliconFlow bge-m3 不支持 `dimensions` 参数，传了报400 |
| bge-m3 向量维度？ | **显式设置 1024** | mem0 默认 1536，不设会导致 Qdrant 集合维度不匹配 |
| DeepSeek base_url？ | **`https://api.deepseek.com/v1`** | OpenAI SDK 自动追加 `/chat/completions` |
| mem0 方法 async/await？ | **同步方法** | mem0 原生是同步的，不用 async |
| mem0 不可用时？ | **抛 `ServiceDegradedError`** | 上层可捕获走降级（暂存原始记录/关键词检索） |

## 十四、配置优先级

```
代码默认值  ←  config/settings.yaml  ←  .env 环境变量
（最弱）                                （最强）
```

`settings.yaml` 中用 `${VAR_NAME:-default}` 引用环境变量：

```yaml
database:
  host: "${DB_HOST:-localhost}"   # 先从 env 取 DB_HOST，没有则用 localhost
llm:
  config:
    api_key: "${DEEPSEEK_API_KEY}"  # 无默认值，必须从 .env 提供
```

`.env` 文件在 `config.py` 模块导入时通过 `load_dotenv()` 加载到 `os.environ`。

## 十五、请求生命周期

以 `POST /api/v1/agent/register` 为例：

```
HTTP Request
  │
  ▼
LoggingMiddleware.dispatch()
  ├─ 生成/提取 X-Trace-Id
  ├─ 记录 start_time
  └─ call_next()
       │
       ▼
AuthMiddleware.dispatch()
  ├─ 提取 X-API-Key / Authorization
  ├─ API Key → SHA256哈希
  └─ call_next()
       │
       ▼
FastAPI Router
  ├─ 路径匹配: POST /api/v1/agent/register
  ├─ Pydantic 校验 AgentRegisterRequest
  └─ Depends(get_db) → AsyncSession
       │
       ▼
agent_register()
  ├─ [Phase 1] 生成 agent_id, api_key
  ├─ [Phase 1] T_AGENT INSERT
  └─ return ok({agent_id, api_key})
       │
       ▼
LoggingMiddleware ← 回到中间件
  ├─ 计算 elapsed_ms
  ├─ 写入日志 (trace_id, method, path, status, ms)
  └─ Response + X-Trace-Id header
       │
       ▼
HTTP Response
```
