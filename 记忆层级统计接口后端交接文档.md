# 记忆层级统计接口后端交接文档

## 1. 文档信息

- 交接日期：2026-07-19
- 目标后端：http://120.27.207.238:8000
- 前端页面：`/memory`，标题“通用记忆建模与多层记忆管理”
- 前端调用目标：`GET /api/v1/memory/stats`

## 2. 当前线上情况

2026-07-19 已通过线上 OpenAPI 和只读请求确认：

- `POST /api/v1/memory/list` 支持 `user_id`、`scene_id`、`task_id` 和分页；单页上限为 100。
- `GET /api/v1/admin/stats` 只返回全局 `total_memories`、`total_users`、`total_agents`、`total_sessions`。
- `GET /api/v1/memory/stats?user_id=user_001` 当前返回 HTTP 404。
- 当前没有面向指定用户、基于完整记忆集合的层级分布接口。

`/api/v1/admin/stats` 不能用于该页面：它是全局管理口径，不受当前 `user_id` 约束，也没有任务级记忆和四层记忆数量。

## 3. 问题说明

前端原先将四张卡片写死为 32%、26%、24%、18%。前端只能拿到当前分页的记忆明细，不能据此计算完整分布。

现有字段 `user_id`、`session_id`、`task_id`、`agent_id` 也不能直接作为互斥分类依据：一条任务记忆可以同时带有用户、会话、任务和智能体标识。其中 `agent_id` 表示产生或使用记忆的智能体，不等于“智能体级记忆”。

因此需要后端提供统一、可追溯的记忆层级定义，并在完整数据范围内聚合。

## 4. 目标与统计口径

### 4.1 层级定义

新增规范字段 `memory_scope`，仅允许以下值：

| 值 | 中文名称 | 语义 |
|---|---|---|
| `user` | 用户级记忆 | 跨会话、跨任务长期复用的用户偏好、稳定事实、习惯和约束 |
| `session` | 会话级记忆 | 只在特定会话或会话延续范围内使用的摘要和上下文 |
| `task` | 任务级记忆 | 任务目标、进展、待办、执行过程和结果 |
| `agent` | 智能体级记忆 | 智能体自身能力、工作流程、运行状态和可跨用户复用的执行经验 |

`memory_scope` 表示记忆的主要复用边界，必须互斥。`memory_type` 表示内容类型，`user_id/session_id/task_id/agent_id` 表示关联对象；这些字段相互独立，不能混用。

### 4.2 统计范围

- `user_id` 必填，统计该用户可见的全部记忆。
- `scene_id` 可选，用于按场景缩小范围。
- 复用 `/memory/list` 的可见性和软删除规则，确保统计总数与列表 `total` 口径一致。
- 默认排除 `status=deleted` 的记录；如果列表还排除了其他状态，统计接口必须使用相同条件。
- `total` 必须等于四个层级 `count` 之和。
- `ratio=count/total`，取值范围为 0 到 1；当 `total=0` 时四个比例均返回 0。
- 无论某层级是否有数据，都必须返回四个层级，缺失层级的 `count` 和 `ratio` 为 0。

## 5. 接口契约

### 5.1 请求

```http
GET /api/v1/memory/stats?user_id=user_001&scene_id=memory-console
X-API-Key: <沿用现有鉴权方式>
Accept: application/json
```

参数：

| 参数 | 位置 | 必填 | 说明 |
|---|---|---|---|
| `user_id` | Query | 是 | 当前用户唯一标识 |
| `scene_id` | Query | 否 | 场景过滤条件 |
| `X-API-Key` | Header | 按现有规则 | 必须沿用记忆列表接口的身份和租户校验 |

### 5.2 成功响应

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "total": 136,
    "level_distribution": [
      { "level": "user", "count": 42, "ratio": 0.3088 },
      { "level": "session", "count": 35, "ratio": 0.2574 },
      { "level": "task", "count": 39, "ratio": 0.2868 },
      { "level": "agent", "count": 20, "ratio": 0.1470 }
    ],
    "generated_at": "2026-07-19T10:00:00Z",
    "classification_version": "memory_scope_v1"
  }
}
```

字段约束：

| 字段 | 类型 | 约束 |
|---|---|---|
| `total` | integer | 大于等于 0，等于四项 `count` 之和 |
| `level_distribution` | array | 固定包含 `user/session/task/agent` 四项 |
| `level` | string | 只能是四个规定值之一 |
| `count` | integer | 大于等于 0 |
| `ratio` | number | 0 到 1；允许浮点舍入误差 |
| `generated_at` | ISO 8601 string | 本次统计生成时间，建议使用 UTC |
| `classification_version` | string | 当前分类规则版本，便于后续迁移和排查 |

### 5.3 错误响应

继续使用现有统一响应包装和错误码：

```json
{
  "code": 40001,
  "message": "user_id is required",
  "data": null,
  "error_code": "INVALID_ARGUMENT",
  "trace_id": "trace_xxx"
}
```

建议状态码：

- 400/422：`user_id` 缺失或参数不合法。
- 401/403：API Key 无效或无权访问该用户数据。
- 500：统计查询异常，响应中保留 `trace_id`。

## 6. 数据模型修改

### 6.1 新增字段

以下 SQL 仅表示目标结构，请根据实际数据库迁移框架调整：

```sql
ALTER TABLE memories
ADD COLUMN memory_scope VARCHAR(16);

ALTER TABLE memories
ADD CONSTRAINT ck_memories_memory_scope
CHECK (memory_scope IN ('user', 'session', 'task', 'agent'));
```

完成历史数据回填后再设置非空约束：

```sql
ALTER TABLE memories
ALTER COLUMN memory_scope SET NOT NULL;
```

建议增加统计索引：

```sql
CREATE INDEX idx_memories_user_status_scope
ON memories (user_id, status, memory_scope);

CREATE INDEX idx_memories_user_scene_status_scope
ON memories (user_id, scene_id, status, memory_scope);
```

### 6.2 历史数据回填

建议采用保守规则：

1. 已存在显式 `metadata.memory_scope` 的记录优先使用该值。
2. 有 `task_id` 的记录回填为 `task`。
3. 没有 `task_id`、但有 `session_id` 的记录回填为 `session`。
4. 其余记录回填为 `user`。
5. 不要因为存在 `agent_id` 就自动回填为 `agent`。`agent_id` 多数情况下只是生产者或调用者标识。
6. 智能体级历史数据只能根据明确的来源类型、元数据或人工确认规则单独回填；没有可靠证据时保持 `user`，并记录回填审计结果。

示意 SQL：

```sql
UPDATE memories
SET memory_scope = CASE
  WHEN metadata->>'memory_scope' IN ('user', 'session', 'task', 'agent')
    THEN metadata->>'memory_scope'
  WHEN NULLIF(task_id, '') IS NOT NULL THEN 'task'
  WHEN NULLIF(session_id, '') IS NOT NULL THEN 'session'
  ELSE 'user'
END
WHERE memory_scope IS NULL;
```

回填前应先统计每条规则命中的数量，并保留迁移日志。若实际 `metadata` 不是 JSONB，请调整取值方式。

## 7. 写入与生成链路修改

以下入口都需要支持并持久化 `memory_scope`：

- `POST /api/v1/memory/write`
- `POST /api/v1/memory/async_write`
- `POST /api/v1/memory/generate`
- `POST /api/v1/memory/generate/batch`
- `POST /api/v1/memory/generate/async`
- 长对话压缩或其他会产生持久化记忆的内部链路

为兼容现有调用方，第一阶段可将请求字段设为可选，并按以下优先级确定：

1. 请求显式传入合法 `memory_scope`：直接使用。
2. 有 `task_id`：使用 `task`。
3. 无任务但有 `session_id`：使用 `session`。
4. 其他情况：使用 `user`。

`agent` 必须显式传入，或由经过评审的智能体状态/流程写入链路设置，禁止仅根据 `agent_id` 推断。

当所有前端和内部调用方完成升级后，可逐步将 `memory_scope` 改为必填字段。

## 8. 统计查询实现建议

统计必须在数据库或服务端仓储层完成，不能先分页再在应用层累加。

示意查询：

```sql
SELECT memory_scope, COUNT(*) AS count
FROM memories
WHERE user_id = :user_id
  AND status <> 'deleted'
  AND (:scene_id IS NULL OR scene_id = :scene_id)
GROUP BY memory_scope;
```

服务层负责：

1. 补齐未出现的层级为 0。
2. 计算 `total` 和 `ratio`。
3. 生成 `generated_at`、`classification_version`。
4. 套用现有 `{code,message,data,trace_id}` 响应结构。

FastAPI 伪代码：

```python
@router.get('/api/v1/memory/stats')
async def memory_stats(
    user_id: str,
    scene_id: str | None = None,
    identity: Identity = Depends(require_identity),
):
    authorize_user(identity, user_id)
    counts = await memory_store.count_by_scope(
        user_id=user_id,
        scene_id=scene_id,
        exclude_statuses={'deleted'},
    )
    levels = ('user', 'session', 'task', 'agent')
    total = sum(counts.get(level, 0) for level in levels)
    distribution = [
        {
            'level': level,
            'count': counts.get(level, 0),
            'ratio': counts.get(level, 0) / total if total else 0,
        }
        for level in levels
    ]
    return success({
        'total': total,
        'level_distribution': distribution,
        'generated_at': utc_now_iso(),
        'classification_version': 'memory_scope_v1',
    })
```

## 9. 权限、性能与一致性

- 必须复用 `/memory/list` 的 API Key、用户隔离和租户权限逻辑。
- 禁止仅信任 Query 中的 `user_id` 而绕过调用者授权。
- 统计条件应抽成与列表共用的仓储过滤器，避免 `stats.total` 和 `list.total` 长期不一致。
- 建议单次 `GROUP BY` 查询完成统计；不要循环发起四次计数查询。
- 数据量较大时可以增加 5 至 15 秒短缓存，缓存键至少包含租户、`user_id` 和 `scene_id`。
- 写入、删除、恢复或调整 `memory_scope` 后应主动失效相关缓存。
- 返回接口耗时和 `trace_id`，便于前后端联调。

## 10. 推荐的列表接口后续增强

这不是四张统计卡片上线的硬前置条件，但建议同步规划：

- `/api/v1/memory/list` 增加 `session_id`、`agent_id`、`memory_scope` 过滤。
- `MemoryItem` 响应增加 `memory_scope`。
- 用户级、会话级、任务级和智能体级子页面使用服务端过滤及响应 `total`，不要用当前页数组长度冒充总数。

## 11. 验收测试

### 11.1 接口测试

至少覆盖：

1. 用户没有记忆：返回四项 0，`total=0`。
2. 四个层级都有数据：计数准确，`total` 等于计数之和。
3. 只有一个层级有数据：其他三项仍存在且为 0。
4. `scene_id` 过滤前后统计正确。
5. `status=deleted` 不计入统计。
6. 同一条记录同时有 `session_id/task_id/agent_id` 时，只按其 `memory_scope` 计入一次。
7. 缺少 `user_id` 返回参数错误。
8. 无权访问其他用户时返回 403。
9. 大数据量下只执行聚合查询，不加载全部明细到应用内存。
10. `classification_version` 固定返回 `memory_scope_v1`。

校验表达式：

```text
total == sum(level_distribution[*].count)
abs(sum(level_distribution[*].ratio) - 1) < 0.001  （total > 0）
```

## 12. 前端接入状态与上线顺序

前端已按本契约调用：

```http
GET /api/v1/memory/stats?user_id=<当前配置用户>
```

接口未上线或请求失败时，页面显示“统计暂不可用”和 `--`，不会再显示固定百分比；记忆列表仍可独立加载。接口上线后无需再次修改页面即可显示真实条数和比例。

推荐上线顺序：

1. 后端增加字段、迁移和回填脚本。
2. 后端更新写入/生成链路。
3. 后端上线统计接口并完成接口测试。
4. 使用真实用户数据核对 `stats.total` 与 `list.total`。
5. 前端在测试环境验证四张卡片、刷新、删除后的统计变化。
6. 观察错误率和查询耗时后再全量发布。
